## REND3R.bat Behavior Summary

The batch file is a local desktop reference tool. It:

- sets `TARGET_DURATION=28800`, which is 8 hours
- defines default video `00.mp4`
- defines MP3 preset names for fire, beach, breeze, cafe, city, crickets, forest, and rain
- prompts for preset or custom mode
- in custom mode, asks for MP4 filename, MP3 filename, and output base name
- loops video and audio inputs indefinitely
- maps video stream first and audio stream second
- copies video codec, encodes audio to AAC, limits duration, and uses faststart

Core command shape:

```bat
ffmpeg ^
  -stream_loop -1 -i "%V%" ^
  -stream_loop -1 -i "%A%" ^
  -map 0:v:0 -map 1:a:0 -c:v copy -c:a aac -t %TARGET_DURATION% -movflags +faststart ^
  "%OUT%"
```

Required correction for Rand0m:

- Default duration must be `8` seconds, not `28800` seconds.
- The web UI should never default to an 8-hour render.
- Output filename must be validated before any local or worker render starts.

## Safe Implementation Options

### Option A: Local Desktop/Dev-Only FFmpeg Tool

Best first execution path for development validation.

Pros:

- uses proven FFmpeg command shape
- avoids browser constraints
- avoids cloud cost
- easy rollback

Cons:

- not available to hosted web users
- Windows-path assumptions must be removed
- binary distribution and trust must be handled outside Git

Recommended for:

- local proof scripts
- fixture generation
- manual exports during development

### Option B: Firebase/Cloud Run Render Worker

Best production path after UI/contract are proven.

Pros:

- server-side FFmpeg boundary
- avoids exposing binaries in the browser
- supports quotas and authenticated jobs
- can write outputs to controlled storage later

Cons:

- cost and quota risk
- upload size and storage lifecycle risk
- needs auth, rate limits, job status, cleanup, and abuse controls

Recommended for:

- future account-based production rendering
- short bounded render jobs only

### Option C: ffmpeg.wasm

Not recommended for MVP.

Pros:

- client-side only
- no server job infrastructure

Cons:

- heavy download
- memory/CPU risk
- poor mobile fit
- browser compatibility complexity
- weak fit for large MP4/audio inputs

Recommended for:

- later spike only if tiny clips and strict limits are acceptable

### Option D: Flutter Web Native Media APIs

Not sufficient for the desired behavior.

Pros:

- lightweight previews

Cons:

- cannot reliably mux MP4 + MP3 into exported MP4 across browsers
- cannot replace FFmpeg-grade processing

Recommended for:

- preview only

## Security, File-Size, Quota, and Cost Risks

Never do in browser/client:

- expose unrestricted render execution
- assume client can mux/export large MP4 safely
- allow path-like output names
- autoplay sounds without consent
- upload unlimited files

Future worker rules:

- require auth
- strict duration limit
- strict input size limit
- allowlist preset assets
- scan/sanitize uploaded filenames
- cache or expire outputs
- rate-limit per user
- log job metadata without raw secrets or private file contents
- never use shared unlimited provider-style keys
