# ADR 0006 — Bring Your Own Key (BYOK)

**Date:** 2026-06-16 (session 43 — updated)
**Status:** Decided — `5baba7b` merged to main; additional gates added (≥1-key gate, unkeyed providers disabled)
**Deciders:** Owner + Fable

---

## Context

`rand0m.ai` makes AI provider calls (Anthropic, etc.) on behalf of users. The default path uses a first-party provider key managed by the owner. BYOK allows users to supply their own AI provider key so their usage is billed to their own account rather than the platform.

This is both a cost-control lever (high-volume users self-fund) and a user-agency feature (power users who already have API contracts).

## Decision

### Key routing
All AI provider calls go through a **Firebase callable proxy** (never directly from the client to the AI provider). The BYOK key is passed to the proxy at call time — it never appears in the compiled client bundle or in plain-text logs.

### Client-side key storage
- **Now:** key stored in Hive on device (unencrypted, local-only). Acceptable for the initial BYOK launch.
- **Later (fast-follow):** key stored encrypted in Hive (symmetric encryption; encryption key derived from device/auth identity). No server-side key storage in v1.

### Key source model

Every AI call carries a `keySource` field:

| Value | Meaning |
| --- | --- |
| `firstParty` | Platform key used; usage billed to platform. |
| `userProvided` | BYOK key used; usage billed to user's account. |
| `missing` | No key available; call blocked with user-facing prompt to add key or upgrade. |

The callable resolves `keySource` at call time based on: (1) whether a BYOK key is present and valid-format, (2) the user's entitlement tier.

### ≥1-key gate (binding)

Before any AI provider call can succeed, at least one valid key source must be resolvable:
- `firstParty` key available (platform-managed) **OR**
- `userProvided` BYOK key present and valid-format.

If neither is available → `keySource=missing` → **provider disabled** for that call. Not just quota-limited — the call is blocked entirely with a user-facing prompt.

**Unkeyed providers disabled:** AI provider integrations that have neither a firstParty key nor a user-provided key are disabled at the entitlement layer. This applies per-provider — a missing Anthropic key does not disable an available OpenAI BYOK key (and vice versa).

### Entitlement interaction
- **Free tier:** platform key with quota cap. BYOK allowed (opt-in; removes cap).
- **Pro/org tier:** platform key with higher cap. BYOK allowed (opt-in; no cap).
- `keySource=missing` surfaces a modal prompting the user to add a key or upgrade.
- Users with no firstParty key and no BYOK key see all AI features disabled (not degraded).

### What is NOT stored
- Keys are never written to Firestore, Firebase Storage, or any server-side system in v1.
- Keys are never logged in Cloud Functions logs.
- Keys are never included in error payloads returned to the client.

## Consequences

- Reduces platform AI cost exposure for high-volume users.
- Requires callable to forward the key to the AI provider on each call — adds one hop but keeps the key off the client bundle.
- Encrypted Hive storage is deferred; unencrypted local storage is acceptable for launch given key-in-proxy routing (key-at-rest risk is local-device-only).
- `keySource` field in telemetry enables cost attribution and abuse detection per-user tier.
