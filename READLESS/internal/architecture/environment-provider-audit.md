# Environment and Provider Configuration Audit

## Status

This audit documents current environment loading and provider configuration
before any consolidation work. It intentionally does not include `.env` values,
Firebase changes, provider routing changes, or runtime behavior changes.

The local master environment direction is to use app-prefixed keys:

| App | Prefix |
| --- | --- |
| `rand0m` | `RANDOM` |
| `up10ad` | `UPLOAD` |
| `out1ine` | `OUTLINE` |
| `knight1y` | `KNIGHTLY` |

Current app code still expects a mix of legacy unprefixed keys, `envied`
generated accessors, `flutter_dotenv`, Dart compile-time environment values,
and Firebase Functions `process.env` values. Phase BW adds shared runtime key
planning metadata in `rk_ai`, but keeps value loading and provider routing
app-owned.

## Provider and Model Configuration

`rk_ai` is already the shared source for provider IDs, provider labels, model
IDs, default models, and model lookup helpers. It defines:

- Google AI / Gemini
- OpenAI
- Claude
- Grok

`knight1y` consumes `rk_ai` from agent configuration and app API services.
Agent provider and model selections are persisted as provider labels and model
IDs, then normalized through shared `rk_ai` helpers.

`rand0m` still owns local provider constants and local API request logic for
Google AI, OpenAI, Claude, and Grok. Its Google AI and OpenAI key access is
generated by `envied`; Claude and Grok require compile-time environment values.

`out1ine` and `up10ad` currently carry `envied` declarations and
`google_generative_ai` dependencies, but this audit did not find live AI
provider routing in app code.

environment prefixes without changing behavior:

| App | Current app identity | Required prefix |
| --- | --- | --- |
| `rand0m` | public/root chat app | `RANDOM_*` |
| `up10ad` | creative render/upload tool | `UPLOAD_*` |

| Google PageSpeed | `GOOGLE_PAGESPEED_API_KEY` |
| Weather / OpenWeather | `WEATHER_API_KEY` |
| Claude | `CLAUDEAI_API_KEY` |

- `RANDOM_OPENAI_API_KEY`
- `UPLOAD_GOOGLEAI_API_KEY`
- `OUTLINE_WEATHER_API_KEY`
- `KNIGHTLY_GROKAI_API_KEY`

The consolidation layer should decide whether to support legacy unprefixed
fallbacks during migration. If fallbacks remain temporarily, they should be
documented as compatibility-only and removed after all apps use prefixed keys.

## Shared Contract Prep

Phase AX adds secret-free key metadata to `rk_core`:

- `EcosystemEnvironmentKeyType`
- `EcosystemEnvironmentKey`
- `EcosystemEnvironment`

These contracts derive key names only. They do not read `.env`, inspect process
environment values, initialize Firebase, or route provider calls.

The shared helper mapping is:

| `EcosystemAppId` | Prefix |
| --- | --- |
| `rand0m` | `RANDOM` |
| `up10ad` | `UPLOAD` |
| `out1ine` | `OUTLINE` |
| `knight1y` | `KNIGHTLY` |

The shared key types are:

| Key type | Legacy suffix |
| --- | --- |
| `openAi` | `OPENAI_API_KEY` |
| `googleAi` | `GOOGLEAI_API_KEY` |
| `googlePageSpeed` | `GOOGLE_PAGESPEED_API_KEY` |
| `weather` | `WEATHER_API_KEY` |
| `claudeAi` | `CLAUDEAI_API_KEY` |
| `grokAi` | `GROKAI_API_KEY` |

Future app adapters can call `EcosystemEnvironment.keyNameFor(...)` or
`EcosystemEnvironment.keysForApp(...)` to derive names such as
`RANDOM_OPENAI_API_KEY` while preserving legacy fallback names from
`EcosystemEnvironmentKey.legacyName`.

Phase AY adds provider configuration metadata to `rk_ai`:

- `ProviderConfigurationKind`
- `ProviderKeyRequirement`
- `ProviderConfiguration`
- `ProviderConfigurations`

These contracts connect provider/service metadata to `rk_core` environment key
types. They describe display names, key requirements, AI-provider mappings, and
known default models without reading values or changing provider routing.

| --- | --- | --- | --- |
| OpenAI | AI | `openAi` | yes |
| Google AI | AI | `googleAi` | yes |
| Google PageSpeed | Diagnostics | `googlePageSpeed` | no |
| Weather / OpenWeather | Weather | `weather` | no |
| Claude | AI | `claudeAi` | yes |
| Grok | AI | `grokAi` | yes |

## Duplicated Configuration Code

Duplicated or scattered patterns found:

- `lib/env/env.dart` exists in every app with similar `envied` declarations.
- `.env` is listed as an asset in every app.
- `flutter_dotenv` and `envied` are both present in every app, but startup
  loading differs by app.
- `Environment` Appwrite metadata files still contain old project naming in
  some apps.
- `rand0m` owns local provider/model constants that overlap with `rk_ai`.
- `knight1y` has app-local API wrappers that now partially delegate provider
  and model identity to `rk_ai`.
- PageSpeed reads a raw `dotenv.env` key instead of the generated env accessor
  pattern used elsewhere.
- Firebase Functions use server-side `process.env` provider keys independently
  from client app env loading.

## Missing Provider Slots

Current gaps to resolve before consolidation:

- `rand0m` has no generated `Env` accessors for Claude or Grok, even though
  its provider UI and service can call both.
- `rand0m` uses compile-time values for Claude and Grok while Google AI and
  OpenAI use generated `.env` accessors.
- `out1ine` and `up10ad` declare only Google AI and OpenAI accessors; they do
  not currently declare Weather, Claude, Grok, or PageSpeed slots.
- `knight1y` does not expose PageSpeed through `Env`; the secret/test widget
  reads `dotenv.env` directly.
- No app currently documents prefixed master-env mapping in code.
- No shared contract currently maps `EcosystemAppId` to environment key prefix.

## Safe Future Consolidation Path

1. Add pure Dart environment key metadata to a shared contract package, likely
   `rk_core` or a future lightweight env/config contract module. This should
   define app prefixes and provider key suffixes only, with no secret values.
2. Add app-local adapter classes that resolve prefixed keys first and legacy
   unprefixed keys second. Keep the adapters in app code at first so runtime
   behavior stays unchanged.
3. Update `envied` declarations per app to include the prefixed keys needed by
   that app. Do not remove legacy unprefixed keys in the first write phase.
4. Normalize `rand0m` Claude and Grok access so all provider keys use the same
   app-local env adapter instead of mixing `Env` accessors and
   `String.fromEnvironment`.
5. Add a generated or adapter-backed accessor for Knight1y PageSpeed so the
   secret/test surface no longer reaches into `dotenv.env` directly.
6. Keep Knight1y Firebase Functions server-side. Consolidate only the key name
   mapping and provider/model contracts; do not move provider secrets into
   client apps.
7. Move provider/model catalogs and defaults fully behind `rk_ai` where app
   services still duplicate constants.
8. After every app resolves prefixed keys successfully, remove legacy
   unprefixed fallbacks in a dedicated cleanup phase.

## Non-Goals For This Phase

This phase did not:

- read or print real `.env` values
- modify `.env` files
- change Firebase/auth configuration
- change provider routing
- change runtime behavior
- edit hosting or production config
- alter app source code
